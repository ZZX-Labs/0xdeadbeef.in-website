Project Standards

Every project should disclose its real status and operating limits.

State whether the project is active, completed, planned, experimental, archived, private, or restricted

Future concepts should not be represented as completed deployments.

Identify the problem and intended user

The project should explain what it is designed to solve and under what conditions it is useful.

Document architecture and dependencies

Core components, interfaces, data flows, required hardware, software, services, and trust boundaries should be recorded.

Preserve revision history

Changes, deprecated approaches, failures, releases, and design decisions should remain traceable.

Publish limitations

Experimental status, missing features, security constraints, legal restrictions, and unresolved risks should be stated clearly.

Separate public documentation from sensitive implementation detail

Security-sensitive, privacy-sensitive, proprietary, controlled, or operationally dangerous details may require restricted disclosure.

Status Definitions

How project status should be interpreted.

Active

Currently receiving direct development, testing, documentation, research, or design work.

Ongoing

A long-term program progressing intermittently according to available resources and priorities.

Experimental

An early-stage system or method that has not reached stable or validated release status.

Planned

A defined concept awaiting funding, time, hardware, collaborators, data, facilities, or technical prerequisites.

Archived

Completed, inactive, superseded, deprecated, or historically significant work preserved for reference.

Restricted

Work whose source, data, implementation, or operational details cannot be released fully.
