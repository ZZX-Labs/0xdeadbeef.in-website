Principles

Build systems that preserve agency, withstand scrutiny, and remain useful beyond their creator.

These principles govern research, engineering, Bitcoin development, cybersecurity, consulting, publishing, design, creative practice, and institutional development across 0xdeadbeef and ZZX-Labs R&D.

Core Principles Mission Philosophy Motto

Core Principles

The standard applied across all work.

1. Preserve user sovereignty

Users should retain meaningful control over identity, data, keys, devices, money, exports, backups, and the ability to leave a system.

2. Build complete systems

Architecture, implementation, testing, documentation, deployment, maintenance, security, recovery, and archival preservation belong to the same project.

3. Design security from the beginning

Trust boundaries, authentication, authorization, key handling, compartmentalization, physical access, logging, rollback, and incident response should be defined before deployment.

4. Preserve evidence and uncertainty

Observed fact, measured value, inference, hypothesis, recollection, opinion, plan, and speculation should remain clearly distinguishable.

5. Prefer inspectable systems

Source code, interfaces, formats, configurations, dependencies, and operating assumptions should be visible to appropriate users and maintainers.

6. Minimize unnecessary dependency

Local operation, offline capability, replaceable components, standard formats, independent hosting, and exportable data should be favored where practical.

7. Document what is built

Useful systems require setup instructions, diagrams, operating procedures, dependencies, limitations, test records, recovery steps, and maintenance guidance.

8. Treat Bitcoin as infrastructure

Bitcoin should be approached through nodes, keys, wallets, payments, mining, Lightning, verification, treasury operations, and education rather than as a branding device.

9. Respect privacy as a boundary

Collect only what is necessary, retain it only as long as justified, protect it appropriately, and avoid turning convenience into surveillance.

10. Use capability responsibly

The ability to build, automate, collect, infer, track, or deploy does not by itself establish that doing so is lawful, necessary, proportional, safe, or ethical.

11. Build for failure and recovery

Systems should assume outages, compromise, operator error, dependency failure, loss, corruption, disconnection, and changing personnel.

12. Preserve institutional memory

Source, decisions, methods, credentials, diagrams, standards, revisions, and operating knowledge should survive beyond any one individual.

Engineering

Principles for software, firmware, hardware, and systems design.

01

Solve the real problem

Do not optimize a superficial symptom while ignoring the underlying operating constraint.

02

Prefer clear architecture

Components, responsibilities, trust boundaries, interfaces, state transitions, and failure modes should be explicit.

03

Make state visible

Users and operators should be able to determine what the system is doing and why.

04

Fail safely

When the system cannot continue correctly, it should degrade predictably rather than silently corrupting data or authority.

05

Prefer modularity

Replaceable components, stable interfaces, and limited coupling improve testing, repair, reuse, and migration.

06

Minimize hidden behavior

Automatic actions, network communication, data collection, updates, and privileged operations should be visible and controllable.

07

Design for maintenance

A system that cannot be repaired, updated, inspected, or transferred is not complete.

08

Test under real conditions

Performance, power, storage, bandwidth, latency, failure, misuse, and environmental limits should be tested beyond ideal demonstrations.

09

Prefer open formats

Data should remain readable and transferable without permanent dependence on a single proprietary tool.

10

Version everything important

Code, firmware, hardware revisions, datasets, documentation, configurations, releases, and schemas should be traceable.

11

Automate carefully

Automation should reduce error and labor without removing necessary judgment, review, or accountability.

12

Preserve an exit path

Users should be able to export, migrate, recover, replace, or discontinue a system without losing essential control.

Research

Principles for evidence, analysis, and publication.

Define the question before collecting the answer

Scope, intended use, assumptions, exclusions, methods, and success criteria should be stated before analysis begins.

Preserve primary material

Original documents, measurements, code, logs, images, recordings, datasets, and field notes should be retained where lawful and appropriate.

Record provenance

Source, author, date, collection method, license, checksum, version, location, and transformation history should be tracked.

Test competing explanations

Research should identify what evidence would support or disprove alternative interpretations.

Publish limitations

Incomplete data, biased samples, unstable dependencies, uncertain measurements, and provisional conclusions should be stated plainly.

Preserve negative results

Failed experiments, rejected designs, incompatible approaches, and null findings may be as useful as successful outcomes.

Make methods reproducible

Scripts, configurations, versions, hardware details, processing steps, and validation procedures should accompany published work when possible.

Revise without erasing history

When evidence changes, the record should show what changed, why it changed, and which version superseded the earlier one.

Security

Principles for bounded trust and defensive resilience.

Least Privilege

Users, services, devices, and processes should receive only the access required for their defined function.

Compartmentalization

A failure in one component should not automatically compromise the entire system.

Strong Identity

Authentication, key management, role assignment, and authority should be explicit and reviewable.

Data Minimization

Do not collect, retain, or replicate information without a justified operational need.

Recovery Before Crisis

Backups, key rotation, rollback, incident response, offline copies, and emergency procedures should exist before failure.

Secure Defaults

The default configuration should minimize unnecessary exposure and privilege.

Visible Logging

Logs should support accountability and diagnosis without becoming an uncontrolled surveillance archive.

Assume Compromise

Architecture should account for stolen credentials, malicious insiders, dependency failures, physical access, and incomplete detection.

Independent Verification

Important identities, releases, addresses, signatures, keys, and system states should be verified through more than one trusted channel.

Controlled Disclosure

Security-sensitive findings should be released according to risk, authorization, remediation status, and public-interest need.

Physical Security Matters

Software controls cannot compensate for uncontrolled physical access to devices, storage, keys, or infrastructure.

No Security Theater

Claims should be supported by architecture, testing, threat models, operational controls, and evidence.

Bitcoin

Principles for Bitcoin systems and treasury operations.

Verify, do not merely trust

Run the relevant infrastructure, inspect transactions, verify addresses, and understand the custody model.

Control the keys

Ownership is conditional when key control depends entirely on another party.

Backups are part of custody

Key material, descriptors, wallet metadata, passphrases, procedures, and recovery tests should be maintained securely.

Separate savings, operations, and experiments

Treasury, spending, development, testing, and public donation funds should not share unnecessary trust or exposure.

Minimize address ambiguity

Addresses should be verified independently, labeled clearly, and used according to defined purpose.

Prefer explicit policy

Multisig, spending limits, approvals, custody roles, emergency procedures, and audit records should be documented.

Respect irreversibility

Bitcoin transactions require deliberate verification because mistakes are generally not recoverable through institutional reversal.

Use Bitcoin as infrastructure, not decoration

Bitcoin integration should solve real settlement, savings, payment, verification, or coordination problems.

Artificial Intelligence

Principles for private, useful, and accountable AI.

User Control

Users should control memory, data sources, permissions, exports, deletion, and model behavior where practical.

Local First

Local or offline operation should be supported when it materially improves privacy, resilience, or autonomy.

Visible Uncertainty

Fluent output should not be presented as verified fact without evidence.

Data Boundaries

Training, retrieval, memory, logging, and external transmission should be explicit and controllable.

Human Accountability

AI should not become a mechanism for hiding responsibility behind automated output.

Replaceable Models

Systems should avoid unnecessary lock-in to one model, provider, or proprietary memory format.

Evidence Before Authority

Model output should support reasoning, not substitute for primary sources, validation, or expert judgment.

Proportional Automation

High-impact decisions require review, context, appeal, and human responsibility.

Privacy

Principles for data, identity, and personal boundaries.

Collect less

  • Request only information necessary for the stated purpose.
  • Avoid collecting identifiers merely because storage is inexpensive.
  • Prefer local processing where remote collection is unnecessary.
  • Use aggregation, minimization, or synthetic data when it can answer the question.

Protect what remains

  • Define retention and deletion rules.
  • Encrypt sensitive data in storage and transit.
  • Restrict access according to role and purpose.
  • Document disclosure, breach, backup, and recovery procedures.

Creative Practice

Principles for art, design, music, writing, and visual systems.

Form communicates values

Typography, color, rhythm, hierarchy, sound, interaction, and physical form shape how a system is understood.

Clarity is not sterility

A precise interface can still possess character, atmosphere, emotion, and identity.

Art is a research method

Drawing, photography, music, writing, and design reveal relationships that technical analysis may overlook.

Preserve authorship and attribution

Sources, collaborators, licenses, references, and transformations should be credited accurately.

Do not imitate without purpose

Style should serve the work rather than replace original thought.

Keep the source material

Original images, recordings, drafts, project files, notes, and exports should be preserved with revision history.

Institutions and Ventures

Principles for companies, laboratories, treasuries, and partnerships.

Purpose Before Scale

Growth should serve the mission rather than replace it.

Authority Must Be Explicit

Ownership, voting, spending, signing, access, hiring, and operational authority should be documented.

Founder Control Requires Responsibility

Strategic control must be matched by accountability, continuity planning, and operational discipline.

Separate Entities by Function

Research, consulting, operations, manufacturing, intellectual property, treasury, and regional activity may require distinct legal and operational boundaries.

Protect Intellectual Property

Ownership, licensing, disclosure, assignment, and use should be defined before collaboration begins.

Use Written Agreements

Scope, deliverables, payment, confidentiality, liability, acceptance, and dispute procedures should not depend on memory.

Build Institutional Memory

Documentation, standards, training, archives, role definitions, and succession preserve continuity.

Distribute Capability

Regional teams, suppliers, laboratories, mirrors, and infrastructure reduce single points of failure.

Maintain Financial Discipline

Budgets, reserves, treasury policy, procurement, compensation, and funding restrictions should be explicit.

Reject Misaligned Capital

Funding that requires surrender of mission, privacy, control, or technical integrity should be declined.

Plan for Failure

Contracts, access, archives, credentials, keys, backups, and authority should remain manageable during dispute, departure, incapacity, or closure.

Build for Continuity

Institutions should remain useful beyond temporary markets, individuals, and technology cycles.

Consulting and Contracting

Principles for professional engagement.

01

Qualify the purpose

Understand the client, authority, problem, jurisdiction, intended use, stakeholders, risks, and available evidence.

02

Define the scope

Specify deliverables, exclusions, assumptions, milestones, dependencies, ownership, licensing, confidentiality, and acceptance criteria.

03

Secure the engagement

Use verified identities, written authorization, appropriate communication, access controls, evidence procedures, and payment terms.

04

Document the work

Provide findings, source files, diagrams, methods, test results, limitations, revisions, and maintenance guidance as agreed.

05

Refuse misaligned work

Unlawful, abusive, deceptive, indiscriminate, exploitative, or unauthorized requests should not proceed.

Refusals

What these principles reject.

  • Opaque lock-in presented as convenience.
  • Surveillance as the default business model.
  • Security claims unsupported by evidence.
  • Data collection without defined necessity.
  • Automation used to hide responsibility.
  • Centralized dependency where local control is practical.
  • False certainty and undocumented conclusions.
  • Destructive capability without lawful purpose and restraint.
  • Growth that requires abandonment of mission.
  • Misrepresentation of plans as completed work.
  • Uncredited use of others’ work.
  • Permanent shortcuts disguised as architecture.
  • Contracts that leave authority or ownership ambiguous.
  • Institutions that separate control from accountability.
  • Technology deployed without regard for foreseeable harm.
  • Power that cannot be audited, challenged, or exited.
Principled refusal

Declining harmful or misaligned work is part of professional competence, not a failure of technical capability.

Review

Principles must be tested against practice.

These principles are not decorative statements. They should be reflected in architecture, documentation, contracts, interfaces, data handling, key management, funding, publication, and operational decisions.

When a project violates a stated principle, the conflict should be identified and justified rather than ignored.

Principles may evolve when experience, evidence, law, technology, or responsibility requires revision.

Changes should preserve a public record of what was altered and why.
