Operating Principles

How the work is approached.

Build complete systems

Architecture, implementation, testing, documentation, deployment, maintenance, security, and archival preservation are treated as parts of the same project.

Prefer user control

Local operation, inspectable source code, exportable data, replaceable components, documented interfaces, and independent ownership are preferred over closed dependency.

Design security from the beginning

Trust boundaries, identity, authorization, key handling, logging, recovery, compartmentalization, physical access, and failure conditions should be considered before deployment.

Preserve evidence and uncertainty

Verified fact, observation, inference, hypothesis, recollection, planning, and speculation should remain clearly distinguishable.

Document what is built

Source code alone is not enough. Systems should include setup instructions, diagrams, operating procedures, dependencies, limitations, test records, and maintenance guidance.

Use technology responsibly

Capability does not justify indiscriminate deployment. Lawful authority, proportionality, necessity, user agency, privacy, and harm reduction remain essential constraints.
